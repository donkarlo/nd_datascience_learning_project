    # How a Minimal Transformer Works in Training and Prediction for Time-Series Gaussian Forecasting

    ## What “the simplest transformer” means in practice
    
    A “Transformer” is not one specific model; it’s a **recipe**: represent a sequence as vectors, then repeatedly apply **self-attention** (mix information across time steps) and a **small feed-forward network** (process each time step independently), with stabilizers like residual connections and normalization. The original Transformer was introduced as a sequence-to-sequence architecture that **replaces recurrence and convolutions** with attention mechanisms. citeturn4view0
    
    Two important clarifications, because they directly affect “training vs prediction”:
    
    * In the original paper, many tasks are framed as **encoder–decoder** and the decoder generates outputs **autoregressively** (“one element at a time”), using masking so a position can’t look at future positions. citeturn4view0turn6view0
    * In many **time-series forecasting** implementations (including what your code *looks like* it’s doing), the model is often closer to an **encoder-only** transformer block (or a stack of them) that takes an input window and emits **all future steps at once** (non-autoregressive). That makes training and prediction much more similar: both are mostly a single forward pass; training simply adds weight updates.
    
    Your “simplest transformer” code is actually “simple transformer + probabilistic head”: it predicts **two outputs per target dimension** (mean and log-variance), and trains them with a Gaussian negative log-likelihood (NLL). This is a standard approach for heteroscedastic regression (input-dependent noise). citeturn4view1turn2search5
    
    ## The minimal transformer block in one forward pass
    
    Think of each time step as a “token.” For time series, a token is a vector of raw features (temperature, volume, sensor channels, etc.). The simplest transformer block does these conceptual steps:
    
    1) **Project inputs into a model dimension**
    If your raw input is `x ∈ R^(B,T,F)`, a small linear layer maps it to `d_model` so attention has enough capacity. (In text models this is an embedding layer; in time series it’s often a Dense projection.) The original Transformer also emphasizes that with no recurrence/convolution, you must somehow represent order. citeturn5view0
    
    2) **Add positional information**
    Because attention alone doesn’t know which time step is “earlier,” you add a positional encoding. The canonical “simplest” choice is the sinusoidal encoding:
    `PE(pos,2i)=sin(pos/10000^(2i/d_model))`, `PE(pos,2i+1)=cos(pos/10000^(2i/d_model))`. citeturn5view0
    
    3) **Self-attention mixes information across time**
    Self-attention means the query/key/value come from the same sequence: each time step looks at other time steps and forms a weighted mixture. citeturn0search1
    Mathematically (scaled dot-product attention), the Transformer computes:
    `Attention(Q,K,V) = softmax(QK^T / sqrt(d_k)) V`. citeturn6view0
    The scaling by `sqrt(d_k)` is used to prevent dot products from becoming too large and pushing softmax into tiny-gradient regions. citeturn6view0
    
    4) **Residual connection + LayerNorm stabilize deep stacking**
    The original encoder layer wraps each sublayer with a residual connection and LayerNorm: `LayerNorm(x + Sublayer(x))`. citeturn4view0
    
    5) **Position-wise feed-forward network**
    After attention, each time step independently проходит through the same small MLP (two linear layers with a nonlinearity). citeturn6view0
    
    6) **Final prediction head**
    For your Gaussian forecaster, the head outputs (per time step) a concatenation `[μ, log_var]`.

    In plain words: **attention chooses what past parts of the window matter for each time step**, and the feed-forward part transforms the resulting context into the final features.
    
    ## How your training code works step by step
    
    Your training class does four core things: make `(input_window, target_window)` pairs, run the transformer forward, compute Gaussian NLL, and update weights.
    
    ### Windowing turns time series into supervised learning
    
    You use a sliding window to transform a long sequence into many short training examples. This is a very standard method: create fixed-length inputs and fixed-length targets, optionally overlapping, so the ordering becomes “supervised pairs.” citeturn0search22turn0search10
    
    Conceptually, your generator produces something like:
    
    * Training sequence: `S[0], S[1], …, S[N-1]`
    * Windows:
      `x^(b) = S[t : t+T_in]`
      `y^(b) = S[t+shift : t+shift+T_out]`
      repeated for many `t`, with some overlap
    
    This gives arrays shaped like your checks enforce:
    
    * `input_array`: `(B, T, F_in)`
    * `target_array`: `(B, T, F_out)`
    And your code explicitly assumes `T_in == output_time_steps` in this “vanilla version.”
    
    ### What happens during `model.fit` (your training loop)
    
    You compile the model with:
    
    * **Optimizer**: Adam with your configured learning rate
    * **Loss**: `_gaussian_nll`
    
    Keras’s built-in training loop trains by splitting data into mini-batches and iterating for `epochs`. citeturn7view0turn3search0
    Adam is a common stochastic gradient optimizer with adaptive moment estimates. citeturn3search2
    Loss functions define the scalar objective the model tries to minimize. citeturn3search5
    
    The key “training vs prediction” switch is that training runs layers in **training mode** (e.g., Dropout behaves differently). Keras and TensorFlow explicitly support a `training` argument for this reason. citeturn1search4turn1search2
    
    So for each batch (conceptually):
    
    1) Forward pass: `y_pred = model(x, training=True)`
    2) Compute loss: `L = gaussian_nll(y_true, y_pred)`
    3) Backprop + optimizer update (handled by `fit`) citeturn7view0turn3search19
    
    Finally, you store learned weights: `LearnedParameter(weights=model.get_weights())`.
    
    ## Why the Gaussian NLL loss looks exactly like it does
    
    Your model outputs two tensors packed into one last dimension:
    
    * `μ` (mean)
    * `log_var` (log variance)
    
    and your loss is:
    
    \[
    \mathcal{L} \;=\; \mathbb{E}\left[\sum_{f=1}^{F}\frac{1}{2}\left(\log \sigma_f^2 + \frac{(y_f-\mu_f)^2}{\sigma_f^2}\right)\right] \;(+\; \text{constant}),
    \]
    
    where \(\sigma^2 = \exp(\text{log\_var})\).
    
    ### Where the terms come from
    
    A Normal distribution’s density has the form:
    
    \[
    pdf(x;\mu,\sigma) = \frac{\exp(-0.5(x-\mu)^2/\sigma^2)}{\sqrt{2\pi\sigma^2}}
    \]
    
    so **negative log-likelihood** necessarily includes a squared-error term scaled by variance and a log-variance term from the normalization constant. citeturn2search5
    
    Your implementation:
    * computes `inv_var = exp(-log_var)` (so dividing by variance becomes multiplication),
    * computes `0.5*(log_var + squared*inv_var)`,
    * sums across features, averages across steps and batch.
    
    It also drops the constant `0.5*log(2π)` term, which doesn’t affect optimization because it doesn’t depend on model parameters (this follows directly from the Normal pdf form). citeturn2search5
    
    ### Why predict log-variance, not variance
    
    Predicting `log_var` is a classic stability trick. In heteroscedastic regression, it avoids explicit division by (potentially tiny) variance, and it allows the network to output unconstrained real values while still producing positive variance via exponentiation. citeturn4view1turn2search5
    
    Kendall & Gal show the same structure: a residual term weighted by `exp(-s)` plus a regularizer term `+ s` (where `s = log σ^2`). They also explain why the regularizer prevents the trivial solution of predicting infinitely large uncertainty everywhere. citeturn4view1
    
    Your explicit clamp:
    
    * `log_var = max(log_var, log(eps))`
    
    guarantees `var >= eps`, preventing numerical issues like division by zero (in the equivalent `1/var` form) and exploding gradients. The general motivation (avoid division by zero via log-variance parameterization) is consistent with the stability rationale above. citeturn4view1
    
    ## How prediction works in your code
    
    Your prediction class rebuilds the same TF model from the architecture, loads weights, runs one forward pass, and splits outputs into distribution parameters.
    
    ### Building and loading weights
    
    You do:
    
    1) `self._tf_model = architecture.build_tf_model()`
    2) “Build once” by calling the model on a dummy tensor so variables exist
    3) `model.set_weights(learned_parameters.get_weights())`
    
    That “build-once before set_weights” pattern is common in Keras when you need variables created before loading. Keras also notes that model/layer behavior can differ between training and inference, controlled through the `training` argument. citeturn1search4turn1search3
    
    ### Inference mode vs training mode
    
    You predict with:
    
    * `y = model(x, training=False)`
    
    Keras explicitly documents calling a model with `training=False` when you have layers that behave differently at inference (e.g., Dropout, BatchNorm). citeturn1search3turn1search2
    
    So prediction is:
    
    1) Forward: `y_pred = model(x, training=False)`
    2) Split:
       `mu = y_pred[..., :F_out]`
       `log_var = y_pred[..., F_out:]`
    3) Convert uncertainty parameterization: `var = exp(log_var)`
    `exp(log_var)` is exactly the “log variance → positive variance” mapping used in standard heteroscedastic Gaussian modeling. citeturn4view1turn2search5
    
    At the end, your API returns `(mu_numpy, var_numpy)`. Interpreting these:
    
    * `mu` is your point forecast (expected value).
    * `var` is aleatoric uncertainty (data noise) per time step and feature, if the model is well-specified and well-trained. citeturn4view1
    
    ## Simplifications, assumptions, and common failure modes
    
    Your code is intentionally “vanilla,” but it’s helpful to be explicit about what that implies.
    
    **Equal input/output length is a strong design choice.** Your training checks assume the model input time steps match `output_time_steps`. That works, but it nudges you toward “predict a full window in one shot.” This is different from the original autoregressive decoder pattern (“one element at a time”). citeturn4view0
    
    **Causality and leakage depend on masking and window construction.** In the original Transformer decoder, masking prevents positions from attending to subsequent positions to maintain autoregressive correctness. citeturn6view0
    In time series forecasting, you often want a similar guarantee: a prediction for a future time step should not use information from later future steps. Whether you need explicit causal masks depends on whether your architecture uses self-attention over any “future” positions and how your sliding windows align inputs/targets.
    
    **Variance can become miscalibrated even when mean is good.** Gaussian NLL can incentivize “gaming the loss” by inflating or deflating variance in tricky regimes; the log term prevents infinite variance, but practical optimization pathologies and calibration issues can still happen. This is an active research topic in heteroscedastic likelihood training. citeturn4view1turn0search17
    
    **Numerical stability needs more than eps in some settings.** Using log-variance helps prevent division-by-zero and keeps variance positive. citeturn4view1
    But if `log_var` becomes very large, `exp(log_var)` can overflow; many implementations also clip upper bounds or parameterize variance via `softplus` to limit extreme outputs while staying positive (this is a common inference from the fact that variance must remain in a numerically representable range, and from practical stability discussions around predicting distribution parameters). citeturn4view1
    
    **Training vs prediction is mostly the same computation, plus mode switches.** The transformer forward graph is almost identical; what changes is:
    * training performs weight updates via `fit`’s optimizer loop, citeturn7view0
    * and layers like Dropout behave differently depending on `training=True/False`. citeturn1search2turn1search4
